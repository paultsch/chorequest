# ChoreQuest — Claude Context

## Project Summary
Rails 7.1 app that lets parents assign chores to children, who earn tokens redeemable for game time. Deployed on Render.com with a PostgreSQL database.

## Tech Stack
- Ruby on Rails 7.1, PostgreSQL
- Tailwind CSS, Stimulus JS, Importmap
- Devise (Parent auth + AdminUser auth)
- Deployed on Render.com via GitHub

## Pyrch Planning
ChoreQuest is a prototype. The production app being planned is **Pyrch** (pronounced like "perch" — rhymes with church). The domain is **pyrch.ai**. The master plan lives at `.claude/pyrch-plan.md`.

## Key Architecture Notes
- Games are static HTML files in `public/games/` (not Rails views)
- `config.public_file_server.enabled = true` is required on Render.com (no NGINX)
- Game sessions use a heartbeat system — JS polls `/game_sessions/:id/heartbeat` every 60s to deduct tokens
- Seeds use `find_or_create_by!` — safe to re-run without wiping data

## QA Sweep — 2026-03-02

### New Bugs Found

- [ ] [CRITICAL BUG - Security]: `GamesController` has no `authenticate_parent!` guard — any unauthenticated user can GET /games, GET /games/new, POST /games, GET /games/:id/edit, PATCH /games/:id, and DELETE /games/:id; add `before_action :authenticate_parent!` for all write actions (new, create, edit, update, destroy); GET index/show can remain public if desired
- [ ] [HIGH BUG]: `ChildSessionsController#create` looks up the child with `Child.find_by(id: params[:child_id])` using a numeric database ID — the child login form exposes a plain text input for "Child ID", meaning any user who knows or guesses another child's integer ID can attempt to brute-force their 4-digit PIN; the child ID should not be guessable; replace with a lookup by name or a scoped dropdown so the ID is never a user-entered value
- [ ] [HIGH BUG]: `ChildrenController#play` checks `where.not(approved: true).exists?` — if a child has NO chores assigned today, `incomplete` is false, so the gate passes and the child can immediately play games without doing any chores at all; the gate should require at least one approved chore: `@child.chore_assignments.where(scheduled_on: today).where(approved: true).exists?` must be true before play is allowed
- [ ] [HIGH BUG]: `PublicController#play` requires ALL today's assignments to be both `completed == true AND approved == true`, but `ChildrenController#play` (used via the parent-facing child show page) only checks `where.not(approved: true)` — the two code paths have inconsistent gate logic; a child accessing via the `/children/:id` parent path can play as soon as self-marking chores `completed`, bypassing parent approval entirely
- [ ] [HIGH BUG]: Dead code block in `ChoreAssignmentsController#create` — lines 155–171 are unreachable because a `redirect_to ... and return` is executed on line 153 before them; the `created = []` re-assignment, `parsed.each` loop, and the second set of redirect calls will never run; if the intent was to handle a different branch, it is silently broken
- [ ] [HIGH BUG]: `ChoreAttemptsController` is guarded by `before_action :authenticate_parent!` at the class level, meaning children cannot submit chore attempts through the `/chore_attempts` path used on the parent-facing `children/show` view; this entire flow is only accessible to parents — a child visiting `/children/:id` (via PIN session) who clicks "Mark Done" will be redirected to the parent login page instead of submitting their attempt
- [ ] [MEDIUM BUG]: `TokenTransactionsController#create` validates that `child_id` belongs to `current_parent`, but then builds the transaction with `TokenTransaction.new(token_transaction_params)` using the raw params — the ownership check is effectively bypassed if `token_transaction_params[:child_id]` differs from the validated child's id (they should be the same value, but the code does not enforce it by building from `child.token_transactions.build`); refactor to `child.token_transactions.build(token_transaction_params.except(:child_id))`
- [ ] [MEDIUM BUG]: `ChoreAssignmentsController#update` calls `@chore_assignment.update(chore_assignment_params)` which allows `child_id` and `chore_id` to be changed via a PATCH request — a parent could move an assignment to a different child or swap the chore after creation; `child_id` and `chore_id` should be excluded from the update permitted params
- [ ] [MEDIUM BUG]: `home/index.html.erb` line 64 calls `Child.find(session[:child_id])` without a nil/not-found guard — if the child record has been deleted since the session was set, this raises `ActiveRecord::RecordNotFound` and crashes the home page with a 500; the layout already guards with `Child.find_by` in the nav, but the home view does a hard `find`; replace with `Child.find_by(id: session[:child_id])` and add a nil branch
- [ ] [MEDIUM BUG]: `GameSession` model `before_create` callback sets `ended_at` to `started_at + duration_minutes.minutes` — so every newly created session is immediately marked as ended before the heartbeat system ever runs; the heartbeat checks nothing about `ended_at`, but the session show page and any query filtering on `ended_at: nil` for active sessions will see it as already ended; `ended_at` should start as nil and only be set when the session is actually stopped
- [ ] [MEDIUM BUG]: `ChoreAttemptsController#approve` and `#reject` redirect to `admin_root_path` on failure, but the dashboard is at `/admin` — a parent who visits the dashboard at `/admin/dashboard` (the correct URL) and approves from there will be redirected back to `/admin` (the namespace root, which is also the dashboard), which is fine; however the fallback redirect message "Attempt not found" redirects to `admin_root_path` which is the same as the dashboard, confusing the parent with no error context; use `redirect_back(fallback_location: admin_root_path)` to send the parent back where they came from
- [ ] [MEDIUM BUG]: `public/show.html.erb` confetti fires whenever `notice.present? && @assignments.any? { |a| a.approved == true }` — this means confetti fires on ANY notice (including "Chore submitted for review!") if at least one chore is already approved; confetti should only fire on a specific approval event, not on every flash notice
- [ ] [LOW BUG]: `chore_assignments/new.html.erb` is an unstyled scaffold page ("New chore assignment" heading, no padding, no card) that renders the `_form` partial — this path is reachable via the "+ New Chore Assignment" button on the dashboard; the chore assignment UI should be the scheduler, not this bare scaffold; either redirect `new_chore_assignment_path` to the scheduler or style this page
- [ ] [LOW BUG]: `token_transactions/new.html.erb` is an entirely unstyled scaffold page — a parent who clicks "New transaction" arrives at a page with no Tailwind styling, a raw HTML form, and a plain "New token transaction" heading; this page needs to be styled like the rest of the app or replaced with an inline modal
